      Introduction

      In this part of the project, we will learn how to programmatically manage AWS resources and explore a deployment scenario involving infrastructure provisioning and load-balancing.

      The Scenario

      The Massive Surveillance Bureau (MSB) gathers data from millions of sources. This data is used to provide intelligence to a group of authorized agents. You wish to join this challenging and lucrative organization as a system architect. A system architect at the MSB designs and deploys a distributed network of web servers.

      Timely information is extremely valuable to MSB agents, so performance guarantees must be met when considering system design. Thus, MSB systems must be able to handle a varying load and scale accordingly, while minimizing cost.

      Over the next several weeks, you will experience the career of a system architect at the MSB.

      This week you will attend a technical interview where the MSB hires system architects who meet their requirements. CMU has assigned the 15319/15619 course team to prepare you for this interview.

      All the best!!!

      Building Blocks

      Let us start by introducing the building blocks of this project (aka Project 2).

      The Massive Surveillance Bureau receives traffic from millions of devices at hundreds of distributed web proxies. These proxies then forward data to a large number of data centers for storage and processing. At the start, you get to control one of these clouds (shown below).

      Your cloud receives data and service requests at a proxy/load generator. Your job is to correctly handle these requests by passing them to a data center that comprises of multiple VM instances. These VMs either store the given data for further processing, or provide the information an agent is asking for. A new use case for this cloud will be explained every week.

      Introducing the Load Generator and Data Center Instances

      For each student in this course, the MSB has provided access to a single load generator AMI (ID: ami-1810b270). They also give you an image of the data center VM (ami-324ae85). Both these images are closed, so when you launch them, you will not be able to SSH in. Instead, you will be interacting with the web interfaces on each.

      Please start an m3.medium instance of the ami-1810b270 (Load Generator) and an m1.small instance of ami-324ae85a (Data Center VM). Configure the security groups to allow incoming (HTTP) requests on port 80. If possible, use spot instances.

      Once you launch these instances, make sure you authenticate yourself. All MSB servers (load generators as well as data center instances) use a very secure user authentication mechanism, shown below.

      Please visit http://<your-instance-dns-name>/username on both instances and submit your username (Andrew ID). You need to do this every time you boot any instance in this project. To do this using code, submit a request to the URL: http://<your-instance-dns-name>/username?username=<yourandrewid>

      Using the Load Generator

      Before CMU sends you to take the recruitment test, we want you to understand the amount of load different types of instances can handle.

      To see the amount of load an instance experiences, click on your m1.small instance in the EC2 Web Console, and select the monitoring tab. Click on the tab that says "Enable Detailed Monitoring". This gives you minute-by-minute updates of the resource utilization of your m1.small instance.

      Now you can activate the load generator. To do so please visit the following URL on your load generator http://<your-load-generator-instance-dns-name>/test1/

      Here you should submit your Data Center’s m1.small Instance DNS and also a test identifier, which can be any unique alphanumeric string. The test identifier helps you view details of all tests that you have launched. Click on submit.

      Once again, this HTML form can be submitted using code using: http://<your-load-generator-instance-dns-name>/test1?dns=ec2-54-165-98-22.compute.amazonaws.com&testId=Noodles

      The test identifier allows the tracking of all your tests for this week, and allows you to view the results of running or completed tests.

      Note: The dns and testId parameters can be sent to the server in any order in the URL. That is, specifying dns first and testId second is also valid in the URL.

      When you submit the /test1 Form, the load generator begins sending some traffic to the m1.small instance. To view this load graphically, go to the monitoring tab in the EC2 console. You should note down the CPUUtililzation during this testing period (as the test is probably not synchronized with CloudWatch, use the middle three minutes of the five minute test window). Please click on the small CPUUtilization graph to view the actual Utilization at each minute of the test.

      Each time you launch a test from this URL, it will generate a steady load for five minutes and then stop.

      The MSB measures system performance based on the requests handled by an instance every second (rps). To view the average rps for your test (generated at one-minute intervals), please visit: http://<your-load-generator-instance-dns-name>/view-logs?name=result_<yourusername>_<testid>.txt

      Please note down the average rps for the test (Ignore the average rps for the first minute). Also note down the average CPU Utilization of the m1.small instance from the monitoring tab on EC2.

      Now, launch an m1.medium "data center" instance of type ami-324ae85. Repeat the process:

      1. Log in with your Andrew ID. Activate detailed monitoring for the instance using the EC2 Web Console.
      2. Submit the new DNS (of the m1.medium data center instance) and a unique test id to the load generator’s “test1” URL: http://<your-load-generator-instance-dns-name>/test1/
      3. Note down the results at: http://<your-load-generator-instance-dns-name>/view-logs?name=result_<yourusername>_<testid>.txt

      Finally, repeat this process with an m1.large instance of type ami-324ae85a. Please note down the average rps for the last four minutes of the test. We have reason to believe that the checkpoint quiz may have questions asking for these numbers.

      You should now understand the MSB’s system. You are almost ready attempt the recruitment exam.
