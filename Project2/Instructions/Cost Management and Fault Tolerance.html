      Cost Management and Fault Tolerance

      Learning Objectives

      • Develop programs that manage cloud resources, while dealing with resource failures and meeting budget constraints

      MSB Project Management

      Note: Please check the pinned post on Piazza for any last-minute updates and changes to this project write-up.

      Note: For this project checkpoint, ensure that your program/script tags all resources with the Key: Project and Value: 2.3.

      After lots of hard work (and high rps), you finally made it to the top. Yes, you are now a Project Manager at the MSB! Being a PM is not as easy as you expected (still, it should be easier than being a Junior Systems Architect). You are now expected to deal directly with one of the motivating factors for the cloud-- cost management. There are several factors motivating this push towards cost efficiency.

      Unfortunately the MSB was recently breached (despite asking all users to type in their usernames!). Because of this data breach (code name DownwardNeeds), there is now a lot of oversight on the MSB, and their budget has shrunk considerably. You need to ensure that your data center meets the new budget requirements.

      Due to the data breach, the MSB is changing all keys pertaining to any target’s device logs. By calling the update-password page on a data center, a new key is generated at the backend and is used to re-encrypt all the tracking data about the target. For instance:

      This week, your testing program will call this page to change keys for all tracked devices. This is all automated by the data center instances. All you need to ensure is that you programatically scale up and down according to the load. Before being able to do this, you need to understand some useful techniques to complete the test.

      Simple Notification Service

      The first thing you realize when you become a PM, is that everything runs on emails. Every small change in the configuration of the data center gets recorded somewhere. You need to provide all these emails to the Checkpoint Committee for Point Allotment (this means it should be in your S3 bucket within your zip file to be submitted). Fortunately, the MSB is good at tracking emails. For the data center, a special AWS service called the Simple Notification Service is used for this purpose.

      Amazon's Simple Notification Service (SNS) is used to automatically send you emails regarding Auto Scale events, to keep track of your instances within an Auto Scaling group:

      1. In the AWS Management Console, click on SNS.
      2. Click on Create New Topic, specify a name and description when prompted.
      3. Once the Topic is created, note down the Amazon Resource Number (ARN).
      4. Subscribe to the topic using your email address and confirm the subscription when you receive an email.

      SNS can be configured to notify you of all Scale Out and Scale In activities.

      Fault Management

      Since you are no longer able to afford the best and the fastest equipment, your data center servers have become old and buggy - they tend to crash regularly. The Load Balancer will continue to send traffic to these buggy servers until they are no longer marked as healthy. You need to ensure that most of these requests are not lost by setting up the ELB to detect and respond to failure. You should also be able to pinpoint exactly when each of your data center's instances failed.

      Job Retainment Examination

      Due to budget cuts, the MSB has begun a retrenchment drive. Each project manager is given three chances to demonstrate their ability to build a cost-efficient data center. Each test can take a maximum of 24 hours (including any warm up time). The actual test covers 20 hours.

      A Note on Time Warping
      For the rest of this module, you can assume that 1 hour of time in the write-up corresponds to 5 minutes of MSB-time. Thus a 100-minute test corresponds to 20-hour-long “test period” leaving enough time for warm ups.

      Your costs are based on the instance-hours (one instance-hour is 5 minutes of usage). You may use m1 or m3 small, medium or large instances for this project. You will be charged the corresponding on-demand price. You have to use less than 80 instance-hours during the test to get a full score.

      A Note on Instance-Hours
      An instance-hour is the unit of cost when one m3.medium instance is billed for a period of one hour. For the purposes of our tests, time-warping makes 1 hour of MSB time correspond to 5 minutes of real time. Similarly, if you have three instances that are launched together for 8 minutes, you will be charged for 3 instances * 2 hours = 6 instance-hours.

      The instance-hours for m1/m3.small/medium/large are based on the on-demand prices.

      For e.g. m3.medium costs 7 cents per hour on-demand, whereas m1.medium costs 8.7 cents. Thus using an m1.medium for 1 hour is equivalent to using 1.24 instance hours whereas an m1.small (4.4 cents) corresponds to 0.62 instance hours.

      The primary objective of this test is to maximize the requests per second (rps) while minimizing cost. For every minute of the test, you provide a value to the agents based on the rps. The amount earned per “minute” is based on the lookup table below:

      RPS Value to Organization
      0-500 0.5 cents
      500-1000 1 cent
      1000-1500 2 cents
      1500-1750 3 cents
      1750-2000 4 cents
      2000-2250 6 cents
      2250+ 10 cents

      Once you have created ARNs for SNS using the AWS web console, you can proceed with the steps below.

      In order to pass the Job Retainment Examination, you have to write a program or script that performs the following tasks:

      1. Create and manage an ELB such that:
        • The ELB redirects HTTP:80 requests to HTTP:80 on the instance
        • Record the DNS Name of the created ELB
        • Set up the page /heartbeat?username=<yourandrewid> as the ELB health check page
        • Note: If you remember from last week, a data center instance needs to be “activated” by passing it your username. This step ensures that the ELB passes your username to your data center instances.
      2. Create a Launch Configuration for the instances that will become part of the auto scaling group, with the following parameters:
        • AMI ID: ami-3c8f3a54
        • Instance Type: You can choose from (m1-m3) small/medium/large
        • Detailed Monitoring: enabled
      3. Analyse the traffic pattern and figure out a good rule for Scale Out and Scale In operations. We define the following rules to give you an idea of the various parameters that need to be set. Please note that you may find it difficult to achieve a passing grade using these parameters.
        • Minimum Instance Size: 1
        • Maximum Instance Size: 5
        • Desired Instance Capacity: 1
        • ELB Name: <MyELB> (choose your ELB name, make sure it is alphanumeric)
      4. Create clever Auto Scale Policies that will fit the traffic pattern:
        • Create a ScaleOut policy that automatically adds a single instance to the auto scaling group.
        • Create a ScaleIn policy that automatically removes a single instance from the auto scaling group.
        • Link your Auto Scaling group to your ELB.
      5. Create CloudWatch Alarms that invoke the appropriate policy for the following scenarios. Please note that you may find it difficult to achieve a passing grade using these parameters and we provide them as an example. You can use any metrics you find useful from CloudWatch:
        • Scale out when the group's CPU load exceeds 80% on average over a 5 minute interval.
        • Scale in when the group's CPU load is below 20% on average over a 5 minute interval.
      6. Link the CloudWatch Alarms to the ScaleOut and ScaleIn rules of your Auto Scaling Group.
      7. Link your ARNs to the AutoScalingGroup. There should be separate emails for instance launch and termination.
      8. Create a load generator (ami-f40ba19c) instance of size m3.medium and activate it using the following URL: 
        http://<load-generator>/username?username=<your_andrew_id>

        Note: Do not use any load generator other than m3.medium.

      9. Before launching the test, you may wish to warm up your load balancer by visiting the URL: http://<load-generator-dns>/warmup. Enter your ELB DNS and a testId of your choice here. In case you have forgotten from last week, a testId is a unique alphanumeric string that you should assign to each of your tests, so that you can view the logs later. You may choose to do this step multiple times until you are satisfied that your ELB has warmed up. The instance-hours used in the warm up phase do not count towards your final grade.
      10. Once your ELB is warmed up (by now you should know how to do this), start the test by visiting the URL: http://<your-load-generator-dns>/begin-phase-3
      11. Once again, enter your ELB DNS and a testId of your choice here. This will start a 100-minute long test. You can view the progress of the test using the URL:
      12. Your code should request the following HTTP URL: 
        			http://<your-load-generator-dns>/begin-phase-3?dns=<your-elb-dns>&testId=<your-test-id>
      13. You can view the progress of this test using the page: 
        http://<your-load-generator-dns>/view-logs?name=result_<yourUsername>_<yourTestId>

        The third line of the page will contain a traffic pattern token. You need to enter this token in the checkpoint quiz. It helps us identify you and the traffic pattern used for your test while grading.

      14. You can monitor your ELB using the web console:

        The graph above is for HTTP 2xx requests (successful requests). There are other graphs for failed HTTP requests (non-2xx). If you are seeing a large number of non-2xx pages, it indicates that you have misconfigured your ELB or that you are not scaling fast enough to handle the spikes. It could also mean that the data center VM has gone down, but you are not removing it from the ELB in time.

      15. Instances will fail during this test. Monitor them using the “Unhealthy Hosts” graph of the ELB. Note down how long they take to recover to a healthy state. Think about how you can speed up this process.

        For instance, this shows 3 data center nodes failing simultaneously for 3 minutes. By running the test once, you should have an intuition for how likely a failure is.

      16. The test will give you updates of the rps for the good traffic every minute. You can see them at the URL: 
        			http://<your-load-generator>/view-logs?name=result_<username>_<testId>.txt
      17. You can still view a sample of 100 requests recently received by a data center instance using the URL: http://<any-data-center-instance>/view-logs

        This page updates every 5 seconds and could give you an intuition about what is happening on your system.

      18. At the end of the “100 minutes”, you will be told your average rps for the entire test, the value provided (in dollars) and given a final token to hand in. Please submit these in the OLI checkpoint quiz.

        Your code should terminate all resources (make sure you save the results from the load generator and copy the required graphs from the load balancer). Once terminated, you can calculate the instance-hours either by using the CloudWatch alarms that you used to Scale Out and Scale In, or by using the “Healthy Hosts” chart on the ELB Monitoring View. Your SNS emails will be a very accurate source of information.
